# GitHub Issue Triage

GitHub issue 治理工具集,含两个互补技能 —— 宏观全量排序 + 单点深度分析。

## 🗂️ triage —— 全量优先级排序

把仓库**全部 open issue** 拉下来,按统一维度分级排序,产出可维护的优先级文档,可配合 `/schedule` 定期增量巡检。

- 用法:"分析 BeeCount 的 issues 排个优先级"、"看下还有哪些急需修的 bug"、"每周一帮我 triage 新 issue"
- 分级:🔴 P0 数据正确性/崩溃 · 🟡 P1 高呼声/性能/卖点相关 · 🟢 P2 局部体验 · ⚪ P3 长尾
- 排序权重:**数据正确性 > 呼声(👍/💬) > 与核心卖点相关度**
- 产出:`.docs/issue-triage.md`(分级表 + 主题聚类 epic 视角 + 推荐 roadmap)

## 🔍 analyze-issue —— 单 issue 深度分析

聚焦**一条** issue 做透:是不是真 bug / 要不要修 / 优先级多高 / 功能请求是否合理。

- 用法:"分析一下 issue #123"、"#123 是不是 bug 要不要修"、"这个需求合理吗"
- bug:在代码库定位、验证根因 / 影响面 / 可复现性,区分真 bug 与使用问题
- 功能请求:评估契合度 / 呼声 / 可行性,**诚实给结论**(该说"不建议做"就说),附拆分建议
- 产出:结构化结论(可作 issue 评论草稿)

## 原则

**只读分析** —— 不自动关闭 / 评论 / 改 issue 状态。产出文档 / 结论供人决策。
